# ATLAS on-disk format — spec v0.1

**Status:** Draft, Phase 0
**Scope:** chunks, blob manifests, file manifests, directory manifests, refs, commits, branches
**Non-scope in v0.1:** erasure-coded chunks, remote chunk stores, distributed metadata, lineage edges, policy objects — these extend in later minor versions

This document is authoritative. If code disagrees with the spec, the code is wrong.

---

## 1. Hashes

All hashes are **BLAKE3-256** — 32 raw bytes. A hex-encoded hash is 64 lowercase characters.

```
hash_hex = <64 lowercase hex chars>      // external, stable
hash_raw = [u8; 32]                      // internal
```

## 2. Chunks (T0.4)

- Default chunk size: **4 MiB** (4,194,304 bytes). Last chunk of a blob may be smaller.
- Chunk hash: `BLAKE3(chunk_bytes)`.
- On disk, a chunk is stored unframed — the content is the file itself. No header.
- Local backend layout:

  ```
  <store>/chunks/<hash[0..2]>/<hash[2..4]>/<hash_hex>
  ```

- `verify(chunk)`: recompute hash, compare.

## 3. Canonical serialization

All manifests are serialized with **bincode v1 legacy** (fixed-int, little-endian). This is deterministic enough for hashing when every field is sorted or written in declared order.

- Struct field order is the declared order in the Rust type.
- Maps are serialized as sorted vectors of `(key, value)` pairs.
- Optional fields: `Option<T>` — 0 or 1 byte tag + value.

Every manifest's `hash` field is `BLAKE3(canonical_bytes_with_hash_zeroed)` — i.e. hash over the serialization with the `hash` slot filled with 32 zero bytes. This makes self-verification trivial.

## 4. Blob manifest

```rust
struct BlobManifest {
    hash: [u8; 32],
    total_size: u64,
    format_hint: Option<String>,   // "parquet", "safetensors", "jsonl", ...
    chunks: Vec<ChunkRef>,
}

struct ChunkRef {
    hash: [u8; 32],
    length: u32,                   // must not exceed chunk size; last chunk may be smaller
}
```

## 5. File manifest

```rust
struct FileManifest {
    hash: [u8; 32],
    blob_hash: [u8; 32],
    created_at: i64,               // unix millis, UTC
    mode: u32,                     // POSIX-style permissions; 0o100644 default
    xattrs: Vec<(String, Vec<u8>)>,// sorted by key
    embeddings: Vec<EmbeddingRef>, // empty in v0.1
    schema_ref: Option<[u8; 32]>,  // None in v0.1
    lineage_ref: Option<[u8; 32]>, // None in v0.1
    policy_ref: Option<[u8; 32]>,  // None in v0.1
    signatures: Vec<Signature>,    // empty in v0.1
}

struct EmbeddingRef {
    model_id: String,
    model_version: String,
    vector_ref: [u8; 32],          // opaque reference into the indexer
}

struct Signature {
    signer_id: String,
    algorithm: String,             // "ed25519" reserved
    bytes: Vec<u8>,
}
```

## 6. Directory manifest

```rust
struct DirectoryManifest {
    hash: [u8; 32],
    entries: Vec<DirEntry>,        // sorted by name
    xattrs: Vec<(String, Vec<u8>)>,
    policy_ref: Option<[u8; 32]>,
}

struct DirEntry {
    name: String,                  // no slashes; no ".." or "."
    object_hash: [u8; 32],
    kind: ObjectKind,              // file | dir | symlink | refspec
}

enum ObjectKind {
    File = 1,
    Dir = 2,
    Symlink = 3,
    Refspec = 4,
}
```

## 7. Ref

A **ref** is a mutable pointer from a logical path to an object hash. Refs live in the metadata plane — they are not content-addressed.

```rust
struct RefRecord {
    path: String,                  // absolute, forward-slash, no trailing slash
    target: [u8; 32],              // object hash (dir or file manifest)
    updated_at: i64,
}
```

## 8. Commit

```rust
struct Commit {
    hash: [u8; 32],
    tree_hash: [u8; 32],           // directory manifest hash at root of the commit
    parents: Vec<[u8; 32]>,        // 0 = root commit; 1 = linear; >=2 = merge
    author: Author,
    timestamp: i64,                // unix millis
    message: String,
    signature: Option<Signature>,  // None in v0.1
}

struct Author {
    name: String,
    email: String,
    agent_id: Option<String>,
}
```

## 9. Branch

```rust
struct Branch {
    name: String,                  // valid identifier, no whitespace
    head: [u8; 32],                // commit hash
    protection: BranchProtection,
}

struct BranchProtection {
    require_signed: bool,          // false in v0.1
    require_reviewed: bool,        // false in v0.1
}
```

## 10. Key schema in the metadata KV

Keys are ASCII, colon-separated, lowercase. Values are bincode-serialized.

| Key pattern | Value | Purpose |
|---|---|---|
| `obj:<hash_hex>` | serialized manifest (blob, file, or dir) | object store |
| `ref:<path>` | `RefRecord` | path → object mapping |
| `branch:<name>` | `Branch` | named branch |
| `commit:<hash_hex>` | `Commit` | commit store |
| `head` | `HeadState` | current HEAD (branch name or detached commit) |
| `xattr:<hash_hex>:<key>` | bytes | out-of-band xattr store (redundant with manifest; for fast updates) |
| `config` | `StoreConfig` | store-level config |

```rust
enum HeadState {
    Branch(String),
    DetachedCommit([u8; 32]),
}
```

## 11. Invariants

- A chunk file on disk named `H` must satisfy `BLAKE3(content) == H`.
- A manifest `M` with declared `hash = H` must satisfy `BLAKE3(serialize(M with M.hash = [0;32])) == H`.
- A directory's entries are sorted lexicographically by `name`.
- A file's xattrs are sorted lexicographically by key.
- Every commit's `tree_hash` must exist in the object store.
- Every branch's `head` must exist in the commit store.
- HEAD, if a branch, must point to a branch that exists.

## 12. Migration from v0.1

Future minor versions add fields with `Option<...>` defaults. Major version bumps require an explicit migration via `atlasctl migrate`. Format version is embedded in `StoreConfig.format_version`.

## 13. Binary test vectors

Test vectors live in `docs/spec/vectors/v0.1/` and must match across all implementations. Each vector is a `(name, input_json, expected_hash_hex, expected_bytes_hex)` tuple. Property tests and fuzzers consume these.
